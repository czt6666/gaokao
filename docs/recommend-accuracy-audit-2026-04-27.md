# 推荐准确性审计报告

**日期**: 2026-04-27  
**目标**: `/api/recommend` 接口  
**测试范围**: 20 个 (省份 × 位次 × 选科) 组合，覆盖鼎尖/中段/中下段、新老高考、3+1+2 三种选科组合  
**总样本**: 1,505 条专业级推荐 + 836 条 hidden_gem 标记  
**采集脚本**: `backend/scripts/audit/collect_recommend.py`  
**分析脚本**: `backend/scripts/audit/{analyze_recommend,check_rank_score,check_hidden_gem}.py`  
**原始结果**: `backend/scripts/audit/out/`

---

## 1. 一句话结论

推荐主流程**整体方向正确**（清北、复交在顶尖位次必出现，冲稳保平均概率单调），但有 **5 个严重缺陷**会让用户在关键时刻做出错误决策，必须先修：

| # | 缺陷 | 严重度 | 影响范围 |
|---|------|------|------|
| 1 | 顶尖位次 (rank≤500) 缺「冲/稳」档，全部塞入「保」 | **P0** | 北京 rank=50/500 全部 30 项都是「保」，0 冲 0 稳；高分学生看到清华勾在「保」+99% |
| 2 | `avg_min_score_3yr` 字段名误导且与位次时间维度不一致 | **P0** | 19% 样本（282/1505）分数 vs 位次反查偏差 >25 分；最大偏差 -188 分 |
| 3 | 同一学校跨多个专业被列入「保」，但概率从 1% 跨到 99% | **P0** | 江苏 rank=500 上海交大医学院在「保」档却 prob=4%；上交某专业「保」prob=1% |
| 4 | hidden_gem 把热门 985 反复标成冷门 | **P1** | 中山大学/上海交大/复旦等被标 `gem_type=A/B/C` 共 12% 占比 |
| 5 | 同一 case 内学校 `avg_min_rank_3yr` 漂移 30%-130% | **P1** | 南方科技大学江苏物理：2728 vs 8295（同省同选科） |

## 2. 检测方法

### 2.1 参数组合矩阵（20 个 case）

| 段 | 案例 |
|---|---|
| 顶尖位次 (rank ≤ 1000) | 北京-50/500、广东-300、江苏-500、浙江-1000 |
| 高分中段 (3k–20k) | 北京-3k/8k、广东-10k、江苏-15k、河南-20k、山东-20k |
| 中下段 (60k–80k) | 河南-80k、广东-60k、四川-70k |
| 历史方向 | 北京-3k 历史、江苏-10k 历史、河南-30k 历史 |
| 选科对照 | 江苏-20k × {物化生 / 物化地 / 物生地} |

### 2.2 校验维度

1. **「上不了」检测**: 学生 rank > 学校 avg_rank × 2，且仍报 prob ≥ 30%
2. **「太吃亏」检测**: 学生 rank < 学校 avg_rank × 0.3 但分到「冲」；或学生 rank > avg_rank × 1.2 但分到「保」
3. **位次→分数换算**: 用 2025 一分一段表（`backend/data/rank_tables_2025_export.csv`，45,032 行）按 `avg_min_rank_3yr` 反查理论分数，与 API 的 `avg_min_score_3yr` 对照
4. **冷门可信度评分**: 综合软科 Top30/50、985/211 标识、城市热度、与同 case 中位位次的相对位置、gem_type 与城市/排名的一致性

## 3. 关键发现

### 3.1 P0 — 顶尖位次推荐结构破损

**症状**：北京 rank=50（全省前 50）和 rank=500 两个 case 的推荐结果：

```
[北京-顶尖50]   surge=0  stable=0  safe=30   hidden_gems=0
[北京-高分500]  surge=0  stable=0  safe=30   hidden_gems=0
```

全部 30 项都被分到「保」档，没有「冲」也没有「稳」。展开看 safe 列表：

```
清华大学      物理基础类         prob=91.9%  avg_rank=67
北京医科大学  临床医学(5+3一体化) prob=99.0%  avg_rank=102
四川农业大学  [投档线]          prob=99.0%  avg_rank=144
清华大学      工程物理类         prob=99.0%  avg_rank=171
北京大学      工科试验班         prob=99.0%  avg_rank=171
…
```

问题：
- 用户看到「清华大学 在『保』档 99%」会以为自己稳进清华，实际只是某专业的均位次 67 略低于学生 50 → 学生 rank 比学校均位 *差* 一点点，sigmoid 在 rank_diff 很小的情况下输出接近 99%，是数学产物，但展示成「保底」严重误导。
- 同时把「北京医科大学」「四川农业大学」推给 rank=50 的考生当保底，纯粹浪费推荐位（这种学生不需要四川农大保底）。
- 0 个「冲」+ 0 个 hidden_gem → 顶尖学生看不到任何冲刺目标和冷门机会。

**根因**: `recommend_core.py:1644-1653` 的 gap_rate 窗口逻辑：

```python
if rank <= 2000:
    if _gap_rate < -0.80:   # 只剔除"学校比学生难 >80%"
        continue
else:
    if _gap_rate < -0.20 or _gap_rate > 0.40:
        continue
```

rank≤2000 的特殊处理只剔下界、不剔上界 → 所有比学生学校（清华／北大／复旦／交大）都被算「保」。需要：rank ≤ 500 时仍应给「冲/稳」档，并把 surge/stable/safe 的概率切分线随 rank 平移。

### 3.2 P0 — `avg_min_score_3yr` 字段名误导 + 时间维度不一致

**字段语义偏差**: `recommend_core.py:1508-1509` 实际取的是 *该专业最近一年的 min_score 单点值*，但字段命名 `avg_min_score_3yr` 让前端／用户以为是 3 年平均：

```python
_latest_ms = (_base_recs[0].get("min_score") or 0) if _base_recs else 0
_avg_score = round(float(_latest_ms)) if _latest_ms > 0 else 0
```

而 `avg_min_rank_3yr` 是经过 P4 人口归一化的 5 年指数加权平均位次（`rank_method.py:129-133`）。两者**时间维度不在一个体系上**，无法用一分一段表互相反查。

**实测偏差**（按 2025 一分一段反查）：

| 偏差区间 | 占比 |
|---|---|
| \|diff\| ≤ 5 分 | 41% |
| \|diff\| ≤ 10 分 | 60% |
| \|diff\| ≤ 25 分 | 81% |
| \|diff\| > 25 分 | **19%** (282 条) |

最严重的样本：

```
河南-中3000  山东大学    avg_rank=2674  API_score=509  反查2025=654  diff=-145
河南-中3000  郑州轻工业    avg_rank=2464  API_score=468  反查2025=656  diff=-188
河南-中3000  重庆师范大学  avg_rank=2440  API_score=475  反查2025=657  diff=-182
北京-顶尖50  四川农业大学  avg_rank=144   API_score=549  反查2025=696  diff=-147
```

我直接 dump 了山东大学@河南数据库里全部 199 条 record，分数范围 **574-664**，**没有 509 分**的记录。509 是从某个 (山大, 某 major) group 的最近一条 record 取的（可能是已废弃专业 + 老年份），与归一化后的 avg_rank=2674 不在同一时间维度。

**用户感受**: 看到「山东大学：均分 509，均位 2674」→ 在河南今年一分一段里 509 分 ≈ 70,000 名 vs 2674 名 → 用户被两个数字之间的矛盾彻底搞蒙。

### 3.3 P0 — 「保」档但概率仅 1-7% 的严重错档

广东 rank=300 / 江苏 rank=500 case 出现：

```
[广东-顶尖300]
  safe: 上海交通大学  人工智能(卓越人才班)        prob=4.0%   avg_rank=346
  safe: 上海交通大学  电子信息(IEEE试点班)        prob=7.3%   avg_rank=353
  safe: 上海交通大学  数学类                     prob=4.0%   avg_rank=346
  safe: 复旦大学医学院  口腔医学                  prob=3.4%
  safe: 上海交通大学医学院  临床医学              prob=4.0%

[江苏-高分500]
  safe: 复旦大学  专业组05(化学)                prob=1.0%   avg_rank=347
  safe: 上海交通大学医学院  专业组03(化学)        prob=4.0%   avg_rank=636
```

这些都是 prob<10% 的「保」——保底的定义是「极大概率能进」，而不是「分到 safe 这一组」。原因可能是分类逻辑只看 gap_rate 不看概率，或者分组阈值与概率阈值脱节。

### 3.4 P0 — 同一学校重复出现且专业级 prob 跨度极大

`test_algorithm_brutal.py:layer6_deep_cases` 已经在测试「推荐结果无重复学校」，但实际生产里**同一所学校反复出现**（每个专业一行）：

```
北京-顶尖50：清华大学出现 5 次（不同专业），avg_rank 从 67 到 260
广东-顶尖300：中山大学出现 6+ 次，avg_rank 从 243 到 410
江苏-高分500：复旦大学出现 6+ 次，prob 从 1.0% 到 96.9%
浙江-高分1000：中国农业大学出现 17+ 次
```

用户看到的 30 项推荐里实际只有 4-6 所学校，剩余都是同校不同专业的重复。这个问题在 `layer6_deep_cases` 测试里**已被掩盖**——那个测试只在 rank=8000 北京物理 这一个相对中段的 case 测，没覆盖顶尖位次。

### 3.5 P1 — 跨 case 同校 avg_min_rank_3yr 漂移过大

筛出在 ≥3 个同省同选科 case 中出现的学校，看其 avg_min_rank_3yr 漂移：

```
南方科技大学  江苏/物理  avgs=[2867, 2728, 2728, 8295]   spread=134%
复旦大学      江苏/物化生 avgs=[527,556,611,594,642,347,703,810]  spread=77%
上海交通大学  广东/物化生 avgs=[260…410]   spread=53%
```

南科大 8295 极有可能是大小年噪音 / 招生计划巨变；复旦 347 这种异常低位次几乎是数据错误。这些在前端就是「同一所校的 avg_rank 一会儿 2700 一会儿 8000」，会被家长直接质疑数据可靠性。

### 3.6 P1 — hidden_gem 把热门 985 反复标成冷门

跨 20 个 case 共 836 行冷门标记 → 183 所去重学校。**12% 的 gem 行**（98/836）属于软科 Top30，明显是热门校：

```
中山大学（软科12，985，广州）       7 个 case 标为 gem_type=B
山东大学（软科15，985，济南）       5 个 case 标为 gem_type=B
浙江大学（软科前列，杭州）           4 个 case 标为 type=C
上海交通大学（软科4，985，上海）     广东-顶尖300 标 type=A（"城市冷"）
复旦大学（软科5，985，上海）         江苏-高分500 标 type=C
```

最荒谬的是把上海交大、复旦标为 `gem_type=A`（"城市冷·专业强"），上海是 heat=10 的最热城市，逻辑反了。

按 gem_type 分布看 D 类（学科强排名低）占 50%（417/836）— D 类应该只对软科 100 名以外触发（`hidden_gem.py:533`），但实际上软科前 50 的学校也被标 D 是因为 `rank_2025=0` 时被当作 "未进入软科 200 强"（fallback 错误）。

### 3.7 P1 — 低置信度 + 极端概率断言

92 处「confidence='低' 且 prob ≥ 90% 或 prob ≤ 5%」。例如：

```
北京-顶尖50  四川农业大学  prob=99.0%  confidence=低
北京-顶尖50  北京医科大学  prob=99.0%  confidence=低
```

数据少 → sigmoid 容易被极端 rank_diff 推到 99/1，但 UI 只显示 99% 的数字，置信度低这件事用户感知不到。

### 3.8 P2 — 选科组合差异不显著

江苏 rank=20000 三种选科对照：

```
物化生：surge=2  stable=46  safe=25  gems=20
物化地：surge=2  stable=46  safe=25  gems=20
物生地：surge=2  stable=46  safe=25  gems=20
```

三组选科推荐量完全相同，suggesting 选科过滤可能在某些路径退化为「只要包含物理就一样」。需要进一步抽查具体学校列表是否差异化。

## 4. "冷门是否真冷"的验证方法（部分有效）

用户问：所谓"冷门"如何验证它真冷？由于无法直接访问竞品 API，我们用**学校客观属性**做代理：

**真冷门应满足**（任一项强成立）：
1. 软科综合排名 100 名以后（不在头部热门校）
2. 不是 985 综合大学（除非 gem_type=A 是城市冷）
3. 城市不在「北京/上海/广州/深圳」（除非 gem_type=B 是专业冷门）
4. 该 case 中的 avg_min_rank_3yr 比同 case 中位数靠后 ≥30%（说明同样难度它录取更容易）

按这个评分系统：

| 可信度桶 | 数量 | 占比 |
|---|---|---|
| 高 (≥70 分) | 0 | 0% |
| 中 (40-70) | 584 | 70% |
| 低 (<40) | 252 | **30%** |

**没有任何一所 gem 拿到 ≥70 分** —— 主要是因为多数 gem 学校仍是 985 或 211，城市也不算偏。这本身不一定是 bug（gem_type=B/D 允许在热门校），但用户层"冷门"二字的预期是"录取难度低于其办学实力"，而我们当前算法输出的是"任意维度有折价"。

**真正最可信的冷门**（被多 case 推、不是 985 也不是热城）：

```
中国传媒大学      9 cases  双一流非985, 北京（专业认知折价: 影视方向）
哈尔滨工程大学     9 cases  双一流非985, 哈尔滨（城市冷+船舶专业强）
河海大学          7 cases  211, 南京（水利专业强但综排靠后）
上海理工大学      7 cases  非985非211, 上海（专业冷门）
河南大学          7 cases  双一流非985, 开封（综合性折价）
青岛大学          5 cases  非985非211, 青岛（城市次热）
燕山大学          5 cases  非985非211, 秦皇岛（A类学科+城市冷）
```

这一批是**站得住脚的真冷门**，方向对。问题只在算法把山大/中山/复旦/交大也混进来。

## 5. 修复建议（按优先级）

### P0（影响商业化展示，必须先修）

1. **修复 rank ≤ 500 的冲稳保切档**（`recommend_core.py:1644-1653`）
   - 顶尖位次也要给冲/稳/保三档；当窗口内没有 surge 时拉宽 gap_rate 上界（如 0.5）让清北的相邻档浮上来
   - 切档逻辑改成"按 prob 而不是 gap_rate"——prob<55 → surge, 55-82 → stable, ≥82 → safe，这样能避免上交大医 prob=4% 出现在 safe 的悖论
2. **重命名 + 重写 `avg_min_score_3yr`**
   - 改成 `avg_min_score_2025`（与 rank 字段语义对齐：用 2025 一分一段反查 `avg_min_rank_3yr` 得到的分数）
   - 或同时返回 `recent_score`（最近年原始分）+ `equivalent_score_2025`（按今年一分一段换算的等位分），让用户知道两个是不同概念
3. **同校多专业去重展示**
   - 推荐结果里同一 `school_name` 只保留 1-3 个 top 专业，而不是把每个专业都列成一行
   - 需要重写前端聚合或后端返回结构（建议后端按 school 聚合，专业信息塞进 `available_majors`）

### P1

4. **冷门标签：白名单+黑名单**
   - 软科综合 ≤30 的学校永远不能被标为 `gem_type=A` 或 `gem_type=B`
   - `hidden_gem.py:calc_city_discount_relative` 当 delta ≤ 0 (学校城市比家乡更热) 时直接 return 0，且 type_a 应跳过这种情况
   - rank_2025=0（缺数据）时不应默认走"软科 200 名以外"的 D 类 fallback
5. **跨 case 漂移修复**
   - 把 `avg_min_rank_3yr` 计算改成「同省同选科同学校跨 case 共用同一个值」（一次性预计算，不在每次请求里重算），减少漂移
6. **低置信度 prob 显示**
   - 当 `confidence=低` 时，prob 数字加灰色 + 显示「数据有限」标签，而不是直接显示 99%

### P2

7. **选科差异化校验**
   - 写一个测试：江苏 rank=20000 的三种选科推荐结果应有 ≥30% 的学校不同
8. **`test_algorithm_brutal.py` 扩展顶尖位次 case**
   - 把 layer6_deep_cases 里的 rank=8000 测试，扩展到 rank=50/500/3000 各做一次去重检验

## 7. 与"市面上产品"对标的局限

我没有直接访问蝶变志愿/优志愿/掌上高考等竞品 API 的能力（需要付费账号或抓取风险）。本审计用的是**线下版市面通用算法**：

- **2025 官方一分一段表**作为分数↔位次的 ground truth（可信，因为是教育部认可的官方数据）
- **软科排名 + 985/211 + 双一流**作为学校层次的 ground truth
- **客观地理城市热度**（北上广深 vs 西部省会）作为城市维度

如果要做更彻底的对标，下一步需要：

1. 选 5-10 个典型 case，手工去蝶变志愿/优志愿网页查同样参数的推荐结果，做学校重叠率（Jaccard）和冲稳保比例对比。
2. 对接微信群里高考老师做小规模盲评（每条推荐打 1-5 分）。
3. 报告里目前的"冷门 30% 可疑"这个数字，是按客观属性算出的代理指标，不能直接当成「30% 的冷门是错的」 — 但可以作为内部回归 baseline。
